# Wise Project Constitution — Base (Streamlit)

## Core Principles

### I. Technology Stack (NON-NEGOTIABLE)

All Wise Streamlit projects MUST use the following stack:

- **Framework**: Streamlit (latest stable) with Python 3.11+
- **Server**: Streamlit runs server-side; business logic in `lib/` modules
- **Authentication**: Supabase Auth (JWT verification server-side via `SERVICE_ROLE_KEY`, session management via `supabase-py` + `st.session_state`)
- **Charts**: Plotly (`plotly.express` + `st.plotly_chart`) for all data visualizations
- **Hosting**: Google Cloud Run (Docker, Python 3.11 slim)
- **HTTP Client**: `httpx` for all external API calls
- **Styling**: Wise theme defined in `.streamlit/config.toml` (provided by wise-standards). The Wise palette (primaryColor `#1D4345`, backgroundColor `#F6F7F8`, textColor `#171616`) MUST NOT be overridden. Additional styling via `st.markdown` with custom CSS is allowed but MUST use the Wise color variables.

Deviations from this stack MUST be justified in the plan's Complexity Tracking table and approved before implementation.

**Rationale**: A consistent stack across Wise projects reduces context-switching, enables code sharing, and simplifies onboarding.

### II. Server-Side Security (NON-NEGOTIABLE)

Streamlit runs entirely server-side, but the browser still receives rendered HTML/JS. All external API calls are inherently server-side, however:

- API secrets (client_secret, service role keys, API keys) MUST remain in environment variables or secrets manager, never hardcoded in source.
- Secrets MUST NEVER be stored in `st.session_state` or displayed in the UI.
- Token management (caching, refresh, retry on 401) MUST be handled in `lib/` modules.
- HTTPS MUST be enforced for all external API calls.

**Rationale**: Even though Streamlit is server-side, secrets can leak through session state serialization, error messages, or logging. Strict isolation is required.

### III. Authentication Guard (NON-NEGOTIABLE)

Every project MUST implement an auth guard pattern:

- **Auth guard function** (`lib/auth.py`): Called at the top of every page. Verifies Supabase JWT from `st.session_state`. Redirects to login page if missing or invalid.
- **Login page** (`pages/Login.py`): Authenticates via `supabase.auth.sign_in_with_password()`. Stores tokens in `st.session_state`.

The auth guard MUST be skippable in local dev when Supabase is not configured.

**Rationale**: Every page must independently verify authentication. Without a centralized middleware like Nuxt, each page must call the guard explicitly.

### IV. Observability

Every external API call MUST produce a structured log entry containing: request method + URL, HTTP status code, entity identifier (where applicable), and wall-clock latency in milliseconds.

- Secrets (tokens, API keys, passwords) MUST be redacted from all logs.
- Failures (network error, 4xx, 5xx) MUST be logged at ERROR level with full error context.
- Use Python `logging` module with `getLogger(__name__)` pattern.

**Rationale**: Integration projects bridge independent systems; silent failures are invisible without mandatory structured logging.

### V. Documentation Artifacts (NON-NEGOTIABLE)

Every feature MUST produce the following artifacts via Speckit:

| Artifact | Generated by | Location |
|---|---|---|
| `spec.md` | `/speckit.specify` | `specs/<feature>/` |
| `business-logic.md` | `/speckit.specify` | `specs/<feature>/` |
| `plan.md` | `/speckit.plan` | `specs/<feature>/` |
| `flows.md` | `/speckit.plan` | `specs/<feature>/` |
| `research.md` | `/speckit.plan` | `specs/<feature>/` |
| `tasks.md` | `/speckit.tasks` | `specs/<feature>/` |

Additionally, `/speckit.plan` MUST:
- Create an ADR in `docs/adr/` if the feature introduces a significant architectural decision (new component, technology choice, structural pattern).
- Update `docs/architecture/overview.md` if the feature impacts the global architecture.
- Update `docs/flows/` if the feature introduces or modifies a cross-cutting flow.
- Update `docs/business-logic.md` if the feature introduces transversal business rules.

Additionally, `/speckit.implement` MUST:
- Update `requirements.txt` when a new pip dependency is introduced by the implementation.
- Never add a dependency without using it in the committed code.

### VI. Simplicity

Implement only what is required by current stories. No speculative abstractions, plugin systems, or generics beyond the documented API surface.

- YAGNI: do not model entities or features not yet required.
- A new abstraction layer MUST be justified in the plan's Complexity Tracking table before being introduced.
- Prefer straightforward code over clever patterns.

**Rationale**: Premature generalization increases maintenance burden without immediate value.

## Security Requirements

- API secrets MUST be supplied via environment variables or a secrets manager; hardcoding in source or config is forbidden.
- HTTPS MUST be enforced for all external calls.
- JWT bearer tokens MUST NOT appear in log output, error messages, or version-controlled files.
- Supabase `SERVICE_ROLE_KEY` MUST NEVER be stored in `st.session_state` or exposed to the browser.
- Dependencies MUST be pinned to specific versions in `requirements.txt`.

## Development Workflow

- Each feature MUST have a corresponding branch following `###-feature-name`.
- All pull requests MUST include a Constitution Check confirming no principle is violated.
- Speckit artifacts (`spec.md`, `plan.md`, `tasks.md`, `flows.md`, `business-logic.md`) MUST be generated before implementation begins.

## Governance

This constitution serves as the base for all Wise Streamlit projects. Individual projects MAY extend it with project-specific principles (e.g., domain-specific API rules) but MUST NOT override base principles.

Amendments require:
1. A pull request updating this file with a version bump.
2. A brief rationale in the Sync Impact Report block.
3. Propagation to all active projects referencing this constitution.

**Versioning policy**:
- MAJOR — principle removed, redefined incompatibly, or governance changed.
- MINOR — new principle or section added, or material guidance expanded.
- PATCH — clarification, wording fix, or non-semantic refinement.

### VII. Technology Stack Override — R Shiny (PROJECT-SPECIFIC)

This project is a **POC (Proof of Concept) for heat pump (PAC) optimization**. The base Streamlit stack (Principle I) is overridden for this project with the following justified deviation:

- **Framework**: R Shiny (latest stable) with R 4.x+
- **UI**: `bslib` (Bootstrap 5 theming) for layout and styling
- **Charts**: Plotly via `plotly` R package for all interactive data visualizations
- **Data manipulation**: `dplyr`, `readr`, `lubridate` (tidyverse ecosystem)
- **Tables**: `DT` (DataTables) for interactive tables
- **Hosting**: Google Cloud Run (Docker, `rocker/shiny` base image)
- **Dependencies**: Managed via `renv.lock` (equivalent to `requirements.txt`)

**Rationale**: The PAC optimization domain relies heavily on time-series simulation, statistical modeling, and interactive parameter exploration. R's ecosystem (tidyverse, Plotly bindings, Shiny reactivity) provides superior expressiveness for this type of numerical/analytical POC compared to Streamlit. This deviation is scoped to this POC only and does not set precedent for other Wise projects.

### IX. Security Adaptations for R Shiny

Base security principles (II, III) apply with these R-specific adaptations:

- API secrets MUST be loaded via `Sys.getenv()` from environment variables, never hardcoded.
- Shiny reactive values MUST NOT store secrets.
- For this POC, **authentication is not required** (internal tool, no external users). If the POC evolves to production, Principle III (auth guard) MUST be implemented via `shinymanager` or equivalent.
- HTTPS is enforced at the Cloud Run level.

### X. R Dependency Management

- `/speckit.implement` MUST update `renv.lock` (via `renv::snapshot()`) when a new R package is introduced.
- Never add a dependency without using it in the committed code.
- All packages MUST be declared in `DESCRIPTION` or loaded explicitly in `app.R`.

### XI. Separation of Concerns — Business Logic vs Shiny Logic (NON-NEGOTIABLE)

All business logic (calculations, models, data transformations, KPI computation, optimization) MUST reside exclusively in R6 classes (`R6_*.R`) or pure functions (`fct_*.R`). These files MUST NOT depend on Shiny (`library(shiny)`, `reactive()`, `observe()`, `session`, etc.).

Shiny modules (`mod_*.R`) MUST contain only:

- UI layout and rendering (`renderPlotly`, `renderDT`, `renderText`, etc.)
- Reactive wiring (reading inputs, calling R6 methods, passing results to outputs)
- User interaction logic (validation messages, modal dialogs, navigation)

Shiny modules MUST NOT contain:

- Mathematical formulas or physical models
- Data transformation pipelines (joins, aggregations, pivots)
- KPI or metric calculations
- Optimizer logic or solver calls
- Business rules or domain-specific conditional logic

**Test rule**: Every R6 class and `fct_*.R` function MUST be unit-testable without loading Shiny (i.e., `library(shiny)` MUST NOT appear in their test files).

**Rationale**: Mixing business logic into reactive code makes it untestable outside of a Shiny session, couples domain knowledge to UI framework specifics, and prevents reuse of calculations across modules or in non-Shiny contexts (scripts, reports, API endpoints).
